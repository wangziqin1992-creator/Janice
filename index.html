上传报错

5 个问题
🔍 原因
Excel 中 productId 或 POI ID 列包含了非数字字符，常见情况：中文字符、前后空格、隐藏格式字符（如从飞书复制时带入的特殊字符）。
✅ 处理步骤
  1. 检查 ID 列，确保为纯数字，无空格、无中文、无隐藏字符。
  2. 可将整列单元格格式设为"文本"或"数字"，然后用"分列"功能重新统一格式。
  3. 删除所有说明文字行和示例行，只保留数据行。
  4. 重新上传。
🔍 原因
上传的 Excel 文件保留了模板自带的说明文字行或示例行，系统尝试将这些文字解析为 ID，触发框架校验失败。
✅ 解决方法
删除模板中所有说明行、示例行、表头说明，只保留纯数据行后重新上传。建议每次从原始空白模板重新填写，而不是在旧文件基础上修改。
🔍 原因
Excel 中存在重复的 productId 或 POI ID，系统不允许同一产品在同一池中重复出现。
✅ 解决方法
上传前用 Excel 的"删除重复项"功能检查并去重 ID 列，确保每个 ID 唯一后重新上传。
🔍 原因
系统有两个不同的模板，用途完全不同，容易混淆：
· 产品上传模板：仅用于批量添加产品到池中
· 营销信息编辑模板：用于批量编辑卖点（Selling Point），支持按语言（Locale）填写

用产品上传模板填写营销信息，系统不会识别，因此不生效。
✅ 正确操作
修改营销信息时，需在秒杀池产品列表页下载专用的 "批量编辑营销信息"模板，填写后上传，不能用产品上传模板代替。
✅ 正确填写方式
在 Excel 模板中，将同一 POI ID 下的不同 Resource ID 分成多行填写，每行一个 Resource ID，不要合并在同一行内。

示例：
POI_12345 | Resource_001
POI_12345 | Resource_002
POI_12345 | Resource_003

选品池 / 秒杀池异常

4 个问题
🔍 常见原因（按顺序排查）
  1. 数据同步延迟：从 BD 提报到 Campaign 2.0 显示，通常有 T+1（1-2天）延迟，选品池数据每天凌晨更新，当天提报当天看不到。
  2. 选品规则命中过少：Rule 1 和 Rule 2 是 OR（或）关系，不是 AND 关系。如果设置的城市/星级条件本身匹配产品就少，结果自然少。
  3. 页面内去重：同一活动页面下，多个选品池命中了相同的产品，系统去重后某些池子的显示数量会减少。这是正常逻辑，不是 bug。
  4. 未选 Applicable Region tag：建池时没有勾选对应市场的 Applicable Region 筛选条件，导致产品被过滤。
✅ 处理方法
  1. 先等 1-2 天观察数据是否同步，紧急情况可联系研发手动触发同步。
  2. 检查 Rule 条件设置，适当放宽筛选范围。
  3. 建池时必须勾选 Applicable Region tag（POI 产品选 "for POI Product"，Activity 产品选 "for Activity Product"）。
  4. 如果需要两个条件的交集（AND），将所有条件合并在 Rule 1 中，或在线下处理好名单后通过 Rule 2 手动上传。
⚠️ 数量显示注意
Pool List 页面显示的 Product QTY 数字有已知计算问题，不准确。请点进具体 Pool 内部查看实际产品列表，以实际列表为准。
🔍 原因
秒杀池创建时间早于 "Scene Task" 功能上线时间，导致任务生成时场次信息不完整。
✅ 解决方法
方法一：重新创建该秒杀池。

方法二：删除池内所有产品后重新添加，系统会为新的产品列表生成完整的 Scene Task。
🔍 可能原因（逐一排查）
  1. Hook Resource ID 无数据：接口未返回对应资源数据。
  2. 产品无可售站点：该产品在目标市场（Locale）没有对应的可售信息，未正确提报。
  3. 页面内去重：同一页面多个池子命中相同产品，去重后该产品未出现在当前池。
📞 升级处理
自查后仍无法解决，需提供以下信息联系研发排查：
· 具体的 Resource ID
· 出问题的 页面链接
· 产品所在的 目标市场（Locale）
🔍 注意
同一个页面组件只能用一种方式配置产品，不可混用：
· 方式一：Pool ID（Campaign 2.0 方式）
· 方式二：Campaign ID + Slot ID（Campaign 1.0 方式）

混用会导致产品展示逻辑互相覆盖，出现不可预期的显示问题。
✅ 正确做法
从 1.0 迁移到 2.0 时，同一组件必须全切，不能保留一半旧配置。建议迁移前做完整的组件清单，逐一确认。

POI / Resource ID 匹配问题

3 个问题
🎯 排查 ID 问题前，先备好这三样
1. 具体的 POI ID / Product ID / Resource ID  2. 目标市场 Locale  3. 相关的选品池或秒杀池链接
🔍 常见原因
  1. 未在目标市场提报：该 ID 可能在其他市场（如 KR）有招商数据，但没有在你的目标市场（如 JP）提报，系统查询该市场时自然找不到。
  2. ID 类型不匹配：产品是 Activity 类型，应使用 Product ID 上传，但误用了 POI ID，或反之。
  3. 数据同步延迟：刚提报的数据还未同步到查询层（T+1）。
✅ 排查步骤
  1. 确认该产品是 Hotel/POI 类型还是 Activity 类型，对应使用正确的 ID 字段。
  2. 确认该产品已在目标市场正确提报。
  3. 如果刚提报，等待 1-2 天数据同步后再查。
  4. 仍有问题 → 提供 ID + Locale + 池链接联系研发排查。
✅ ID 对应规则
POI 产品（景点/酒店）：填写 POI ID + Resource ID（两个都需要）

Activity 产品（活动/票务）：只填写 Product ID（不需要 Resource ID）

用错类型会导致系统找不到产品或数据缺失，这是最常见的 ID 相关问题之一。
✅ 状态
此问题为前端展示 bug（列表数据 region 字段缺失），已由研发修复。如果还在报这个错，请截图提供页面链接联系前端同事，确认修复是否覆盖你的环境。

页面展示异常

4 个问题
🔍 原因
商品侧未维护该 Resource ID 的划线价（门市价/原始卖价），导致系统无法正确显示价格对比关系。这是商品数据问题，不是 Campaign 2.0 平台问题。
📞 处理路径
提供具体的 Resource ID，反馈给商品团队核查并补充划线价。可通过研发同事转发至商品团队处理。
🔍 原因
前端组件的多语言配置未覆盖该站点的语言，或配置错误（如 AU 站英文页面的"折扣"标签被错误配置为中文）。
📞 处理路径
提供出现问题的 Foxpage 预览链接，联系前端或翻译配置负责人检查并修正多语言配置。
🔍 原因
GEO 系统未返回这些城市的目标语言翻译值，前端以英文兜底展示，导致语言不统一（非 bug，是 GEO 数据缺口）。
📞 处理路径
联系 GEO 数据团队，提供缺少翻译的城市列表和目标语言，请其补全对应翻译值。
🔍 系统同步逻辑
· 选品池、评级数据：T+1，每天凌晨更新，不是实时的
· Price Monitor 价格数据：设置后次日才有，活动开始后升级为每天 3 次更新
· Product Pool 计算：修改规则后重新计算，产品越多等待越长，完成后 TP 通知
✅ 处理方法
等待 1-2 天自然同步。如果紧急，可联系研发同事手动触发同步任务加速处理。

QA 测试问题

2 个问题
🔍 原因
QA 系统对某些商品底层状态的检查提示不完善，错误信息无法准确展示。此类问题通常涉及商品侧的底层数据，而非 Campaign 2.0 配置问题。
📞 处理路径
  1. 先联系对应 BD 确认商品状态是否正常。
  2. 如 BD 确认无问题，将问题转至 QA 专项排障群，由商品研发同学进行深度排查。
✅ 提问四要素（缺一不可）
  1. 系统链接：出问题的 Campaign 2.0 配置页链接，或 Foxpage 预览链接
  2. 资源信息：具体的 POI ID、Product ID、Resource ID、Pool ID 或 Campaign ID
  3. 问题截图：完整的报错信息截图或页面异常截图
  4. 操作描述：你做了什么操作、期望看到什么结果、实际看到了什么

提供这四样信息，通常能将问题解决时间缩短 50% 以上。

预防性最佳实践

📤

数据准备与上传

  • 上传前确保 ID 列为纯数字，无空格、无中文、无隐藏字符，用"分列"功能统一格式
  • 区分"产品上传模板"和"营销信息编辑模板"，不要混用
  • 上传前用"删除重复项"检查 ID 列，去除重复行
  • 秒杀产品优先在 ANT Mechanism 配置好后再导入，避免手动上传引发时区和 Session 问题
🗂️

选品池配置

  • 建池时必须勾选 Applicable Region tag，确保产品在目标市场可售
  • 理解 Rule 1 与 Rule 2 是 OR 关系，需要交集时在 Rule 1 中合并所有条件
  • 以 Pool 内部实际列表为准，不要依赖 Pool List 页面的 QTY 数字
  • 修改规则后等待 TP 通知计算完成，再配置 Foxpage，避免上线时产品为空
⏱️

时间节点意识

  • BD 提报后等 1-2 天再操作，选品池数据每天凌晨更新（T+1）
  • Price Monitor 设置后次日才有数据，活动前至少提前 1 天设置
  • 新 Theme 申请每周四提交，活动前 2 周以上申请
  • Pool 规则更新后在活动上线前至少 1 天操作，留出计算时间
